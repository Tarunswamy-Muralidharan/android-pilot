Maestro MCP was used to perform UI testing of the JustPass Android app on a physical Motorola G54 device. The testing session covered 8 of 14+ app screens over approximately 25 minutes of active tool interaction.

The core finding: Maestro MCP is fundamentally a screenshot tool with fragile interaction capabilities. The take_screenshot tool is excellent and reliable. Everything else — tapping, scrolling, launching — has significant reliability issues that required constant workarounds.

Of the ~50+ tool calls made during the session:

  • ~30 succeeded on first attempt (mostly screenshots and simple taps)
  • ~12 required workarounds (coordinate taps, ADB fallbacks, swipe instead of scroll)
  • ~8 outright failed (launch_app, scroll direction, scrollUntilVisible, system UI collision)
Critical Issue: The most basic operation — launch_app — failed entirely and required raw ADB commands as a fallback. This undermines the entire premise of Maestro as a mobile testing abstraction layer.

3. Tool-by-Tool Analysis

3.1 take_screenshot

A+
Reliability Grade
15+
Times Called
0
Failures
2-3s

The undisputed star of the Maestro MCP suite. Every single call returned a clear, full-resolution device screenshot. The image quality was sufficient to read all text, verify layouts, check color schemes, and identify loading states.

What it does well:

  • Captures the complete device screen including status bar and navigation bar
  • Image is clear enough to read small text (timestamps, version numbers, fine print)
  • Consistent 2-3 second response time regardless of screen complexity
  • Works during animations, loading states, and transitions
  • No special parameters needed — just call it and get the screenshot

Limitations:

  • Returns a static image — can't capture animations or transitions (just a frozen frame)
  • No way to capture a specific region/element — always full screen
  • No video recording capability (Maestro's gif_creator equivalent doesn't exist)
Verdict: If Maestro MCP were only a screenshot tool, it would be excellent. This is the tool that carries the entire testing experience.

3.2 inspect_view_hierarchy

B-
Reliability Grade
6+
Times Called
0
Failures
3-4s
Avg Response Time

Returns a CSV-formatted dump of the entire Android accessibility/view tree. Every visible element appears with its bounds, text content, accessibility labels, resource IDs, clickable state, and enabled state.

What it does well:

  • Never failed to return results
  • Provides exact pixel bounds for every element — essential for coordinate taps
  • Shows accessibility labels that tap_on can target
  • Reveals the complete UI tree including off-screen elements in scroll containers

Critical problems:

  • Compose elements have no resource-id. Traditional Android Views get IDs like @id/refresh_button. Jetpack Compose elements show up with empty resource-id fields. This means you can't use the most reliable element targeting method.
  • Massive output size. A typical screen dumps 200-400 lines of CSV. For a complex screen like Dashboard with nested cards, it can exceed 500 lines. Parsing this to find one element is tedious.
  • System UI elements mixed in. The dump includes Android system navigation bar elements (Home, Back, Overview buttons) alongside app elements, with no clear separator between system and app layers.
  • No filtering capability. You can't ask for "only clickable elements" or "only elements with text" — you always get the full dump.

Sample output (abbreviated):

bounds,text,accessibilityText,resourceId,clickable,enabled,hintText
[0,60][1080,2280],,,,false,true,
[0,60][1080,120],,,android:id/action_bar_container,false,true,
[16,2088][264,2256],Home,Home,,true,true,    ← App's Home tab
...
[0,2340][360,2400],Home,,com.android.systemui:id/home,true,true,  ← System Home button!
The collision problem illustrated: Both the app's "Home" tab and the system's Home button appear with text=Home. The system element even has a resource-id (com.android.systemui:id/home) while the app element has none. Maestro's tap_on by text will pick whichever it finds first — which was the system button, sending the app to background.

Detailed Failure Log

Failure #1: "Home" tab tapped system Home button

  • Command: tapOn: text: "Home"
  • Expected: Tap app's Home navigation tab
  • Actual: Tapped Android system Home button — app went to background
  • Impact: Lost app state, had to relaunch via ADB
  • Root cause: Maestro's text matching doesn't distinguish system UI from app UI. Both had text=Home
  • Workaround: Used inspect_view_hierarchy to find app tab bounds [55,2080][249,2250], then tapOn: point: "152, 2165"
  • Time wasted: ~45 seconds (relaunch + hierarchy inspection + coordinate calculation)

Failure #2: Resource ID targeting on Compose

  • Command: tapOn: id: "refresh_button"
  • Expected: Tap the refresh button on Dashboard
  • Actual: Element not found error
  • Root cause: Jetpack Compose doesn't generate resource-id in the accessibility tree
  • Workaround: Used tapOn: accessibilityText: "Refresh"
  • Time wasted: ~15 seconds

Failure #3: Ambiguous text on CA Marks

  • Command: tapOn: text: "CA Marks"
  • Expected: Tap the CA Marks navigation tab
  • Actual: Tapped the header text "CA Marks" on the current screen instead of the navigation tab
  • Root cause: Multiple elements had the same text content
  • Workaround: Coordinate tap on the bottom nav tab position

Failure #4: Tapping off-screen element

  • Command: tapOn: text: "Tap for subject-wise details"
  • Expected: Tap the subject attendance card below the fold
  • Actual: Element not found (it was below the visible viewport)
  • Root cause: tap_on only targets currently visible elements, not elements in scroll containers
  • Workaround: Manual swipe down first, then retry tap

Failure #5: Stale element after navigation

  • After navigating back from a screen, previously cached element references were invalid
  • Required re-inspection of hierarchy to get fresh element data
The fundamental problem with tap_on: In a Compose app with standard navigation labels, you're effectively forced to use coordinate taps for anything in the bottom nav bar. This defeats the purpose of semantic element targeting and makes tests fragile across different screen sizes and densities.

The scroll Command

The scroll command only supports a bare invocation (scrolls down). It does not accept a direction parameter, despite this being documented in Maestro's YAML syntax. This means you can only scroll down, never up — unless you use the swipe workaround.

The swipe Command (workaround)

scrollUntilVisible

scrollUntilVisible is supposed to automatically scroll until a target element appears. In practice:

  • Command: scrollUntilVisible: "Tap for subject-wise details"
  • Result: FAILED — "No visible element found" even though the element existed just one scroll below the viewport
  • Likely cause: The command probably scrolls a finite number of times and gives up, or the element detection fails in scroll containers
Impact: Every scroll-then-tap sequence requires three steps instead of one: (1) manual swipe, (2) take_screenshot to verify visibility, (3) tap. This tripled the interaction count for any content below the fold.

The single worst-performing tool. Failed every single time it was called.

Command:  launch_app: com.example.attendancewidgetlaudea
Error:    "Unable to launch app com.example.attendancewidgetlaudea"

Diagnosis steps taken:

  1. Verified package exists: adb shell pm list packages | grep attendancewidget → confirmed installed
  2. Verified launchable activity: adb shell dumpsys package com.example.attendancewidgetlaudea | grep -A1 "android.intent.action.MAIN" → confirmed .MainActivity
  3. Attempted with explicit activity: still failed
  4. Fallback: adb shell am start -n com.example.attendancewidgetlaudea/.MainActivity → worked every time

Possible causes:

  • Maestro may require Maestro Studio or Maestro CLI running as a daemon, not just ADB connectivity
  • The app's package name or launch intent may not match Maestro's expected format
  • The MCP server may not properly translate the launch command to the underlying Maestro binary
This is a dealbreaker. If a mobile testing tool can't launch the app being tested, nothing else matters. Every time the app needed to be relaunched (e.g., after the Home button collision), the entire Maestro abstraction was bypassed via raw ADB.

6. The Five Critical Flaws

Flaw #1: System UI / App UI Collision (Severity: CRITICAL)

When Maestro matches elements by text, it searches the entire accessibility tree — including the Android system navigation bar. Common labels like "Home", "Back", "Settings", "Search" exist in both the system UI and most apps.

There is no way to scope a tap to "app only" — no Z-layer filter, no package filter, no "exclude system UI" flag. This means:

  • Any app with a "Home" tab is at risk of triggering the system Home button
  • Any app with "Back" text could trigger system Back navigation
  • The only safe approach is coordinate taps, which defeats semantic testing

Flaw #2: Jetpack Compose Incompatibility (Severity: HIGH)

Maestro was designed in the era of XML-based Android layouts where every View had a resource-id. Jetpack Compose doesn't generate these IDs. This means:

  • tapOn: id: is completely useless
  • You're limited to text matching (collision-prone) or accessibility labels (only if developer added them)
  • Most Compose elements need Modifier.semantics { testTag = "..." } to be reliably targetable — requiring code changes specifically for Maestro

As Compose adoption grows (it's now Google's recommended UI toolkit), this becomes an increasingly critical gap.

Flaw #3: Broken Scroll API (Severity: HIGH)

The scroll command's direction parameter throws "Unknown Property". The scrollUntilVisible command fails to find elements that exist just below the viewport. The only working scroll method is swipe — which requires manually managing direction (inverted from what you'd expect) and duration.

Flaw #4: launch_app Total Failure (Severity: CRITICAL)

100% failure rate on app launch. The most basic prerequisite for mobile testing. Forces ADB fallback which means Maestro is not self-contained — you need raw shell access as a safety net.

Flaw #5: No Wait/Poll Mechanism (Severity: MEDIUM)

Mobile apps are inherently asynchronous. Data loads from APIs, animations play, transitions happen. Maestro MCP provides no way to wait for a condition:

  • No waitForElement
  • No waitForText
  • No sleep (blocked by tool harness above 2s)
  • No polling mechanism

This makes it impossible to reliably test anything that involves network requests or async loading. You just have to take screenshots and hope the data has loaded.

7. Jetpack Compose Compatibility Analysis

Since JustPass is built entirely with Jetpack Compose, this section specifically addresses how Maestro MCP handles Compose UI elements.

Compose FeatureMaestro SupportImpact
Text composables Readable Text content appears in hierarchy. Can tap by text (with collision risks).
Button/Clickable Partial Clickable state shown in hierarchy. But no resource-id to target specifically.
LazyColumn/LazyRow Poor Off-screen items not in hierarchy. scrollUntilVisible fails. Must manually swipe.
Modifier.semantics/testTag Works If developer adds testTag, it appears in hierarchy. But requires code changes.
Navigation (NavHost) Indirect Can't observe navigation state. Must screenshot to verify current screen.
Dialogs/Popups Partial Dialog content appears in hierarchy when visible. Can tap elements inside. But no way to wait for dialog to appear.
Animations Invisible Cannot verify animations. Screenshots capture a single frame only.
Liquid Glass shader effects Invisible GPU shader effects not visible in accessibility tree. Only verifiable via screenshot (visual inspection).
Pull-to-refresh Manual only Must use swipe DOWN with careful coordinates. No "pullToRefresh" command.
Key insight: Maestro was built for the View system era. Its element targeting model assumes resource-ids exist. For Compose apps, you either add testTag semantics to your code (test-specific code pollution) or fall back to coordinate taps (fragile). Neither is a great option.

11. Recommendations

For Maestro MCP Developers

  1. Fix launch_app — this is table stakes for any mobile testing tool
  2. Add app-scoped element targeting — a scope: "app" or excludeSystemUI: true flag on tap_on would eliminate the #1 failure mode
  3. Fix the scroll direction parameter — this is a regression or documentation mismatch
  4. Add waitForElement / waitForText — essential for testing any app with network calls
  5. Support Compose testTag targeting — allow tapOn: testTag: "home_tab" to target Compose semantics
  6. Add hierarchy filtering — inspect_view_hierarchy: clickable: true to reduce output noise
  7. Fix scrollUntilVisible — should work reliably for elements in LazyColumn/LazyRow

For App Developers Using Maestro MCP

  1. Add testTag semantics to key interactive elements — especially bottom navigation tabs, buttons, and cards
  2. Avoid common text labels for navigation elements — or add unique accessibility descriptions
  3. Always have ADB access as fallback — don't rely solely on Maestro MCP for critical operations
  4. Use coordinate taps for navigation elements — text-based taps are too risky
  5. Take screenshots liberally — it's the most reliable verification method

For AI Agents Using Maestro MCP

  1. Always inspect the view hierarchy before tapping common labels like "Home", "Back", "Settings"
  2. Default to coordinate taps for navigation elements after the first hierarchy inspection
  3. Use swipe instead of scroll for directional scrolling
  4. Take verification screenshots after every interactive action to confirm the expected result
  5. Keep ADB shell commands ready as fallbacks for launch and any Maestro failures

Summary

Maestro MCP is a screenshot-first tool with bolted-on interaction. For visual verification — "does this screen look right?" — it's excellent. For interactive testing — "tap this, scroll there, verify that" — it's a minefield of workarounds.

The tool works best when you treat it as:

  1. A screenshot machine (primary use)
  2. A view hierarchy inspector (for finding element positions)
  3. A coordinate-based clicker (after finding positions from the hierarchy)

It works worst when you try to use it as:

  1. A semantic element interactor (text/id matching is unreliable)
  2. A scroll manager (broken API)
  3. An app launcher (doesn't work at all)
  4. An automated regression suite (no conditions, no waits, no assertions)

For a Jetpack Compose app like JustPass, approximately 40% of all interactions required workarounds. The most dangerous failure — tapping the system Home button instead of the app's Home tab — would be unacceptable in any automated testing pipeline. The workaround overhead consumed roughly 25% of the total testing time.

Maestro MCP is a promising concept with significant implementation gaps. It's usable today for manual visual testing assisted by an AI agent, but it is not ready for reliable, autonomous mobile UI testing.
